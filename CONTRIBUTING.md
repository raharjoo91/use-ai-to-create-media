## Getting Started

### 1. Fork the Repository

```
1. Visit https://github.com/yourusername/ai-powered-k3-health-promotion
2. Click "Fork" button
3. Clone your fork locally
```

### 2. Set Up Your Environment

```bash
# Clone your fork
git clone https://github.com/YOUR_USERNAME/ai-powered-k3-health-promotion.git

# Navigate to project
cd ai-powered-k3-health-promotion

# Add upstream remote
git remote add upstream https://github.com/original-owner/ai-powered-k3-health-promotion.git
```

### 3. Create a Branch

```bash
# Create feature branch
git checkout -b feature/your-contribution-name

# Or for documentation
git checkout -b docs/your-doc-update
```

## Development Workflow

### Step 1: Understand the Context

```
Read First:
├─ README.md (project overview)
├─ Rule.md (project guardrails)
├─ persona.md (AI specialist role)
└─ CONTRIBUTING.md (this file)
```

### Step 2: Make Your Changes

**Content Guidelines:**
```
✓ Follow ISO 7010 standards
✓ Use WCAG 2.1 AA accessibility
✓ Verify all safety information
✓ Cite sources
✓ Use grade 6-8 reading level
✗ Don't include unverified procedures
✗ Don't use copyrighted materials
✗ Don't compromise safety accuracy
```

**File Naming:**
```
Guides:     guides/##-short-name.md
Templates:  templates/category/template-name.md
Examples:   examples/category/example-name.md
Resources:  resources/topic-name.md
```

### Step 3: Test Your Changes

```
Verification Checklist:
☐ Links work correctly
☐ Markdown renders properly
☐ Code examples are tested
☐ Images are optimized
☐ Accessibility is maintained
☐ Safety information is accurate
☐ Language is clear and simple
```

### Step 4: Commit Your Changes

```bash
# Stage changes
git add .

# Commit with clear message
git commit -m "Add: Fire safety poster template for office buildings"

# Use conventional commits:
# Add:    New content
# Fix:    Bug fixes
# Update: Updates to existing content
# Improve: Improvements without new features
# Docs:   Documentation changes
```

### Step 5: Push and Create Pull Request

```bash
# Push to your fork
git push origin feature/your-contribution-name

# Create pull request on GitHub
```

## Style Guidelines

### Markdown Style

```markdown
# Headings: Use # for h1, ## for h2, etc.
## Leave blank line before and after headings

**Bold** for emphasis
*Italic* for definitions
`Code` for technical terms

- Use dashes for lists
  - Indent nested lists with 2 spaces

> Use quotes for important notes

```

### Code Style

```yaml
# YAML for configuration
key: value
list:
  - item1
  - item2
```

```bash
# Shell commands: Use bash
# Comments explain what commands do
```

### Content Style

```
Language: Indonesian or English
Level: Grade 6-8 (Flesch 60-70)
Voice: Active
Tone: Professional, accessible
Format: Clear sections, bullet points
```

### Accessibility

```
Images:
☐ Alt text included
☐ Descriptive filenames
☐ Meaningful without color

Links:
☐ Descriptive link text
☐ No "click here"
☐ Works with screen readers

Structure:
☐ Logical heading order
☐ Tables have captions
☐ Lists are properly formatted
```

## Submission Process

### Pull Request Template

```markdown
## Description
Brief description of changes

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Content update
- [ ] Documentation
- [ ] Translation
- [ ] Other

## Testing
- [ ] Links verified
- [ ] Code tested
- [ ] Accessibility checked
- [ ] Safety info verified

## Checklist
- [ ] Follows contribution guidelines
- [ ] Proper commit message
- [ ] No merge conflicts
- [ ] Ready for review

## Additional Notes
Any additional context or considerations
```

### Review Process

```
1. Automated Checks
   ├─ Link validation
   ├─ Markdown linting
   └─ Accessibility scan

2. Manual Review
   ├─ Content accuracy
   ├─ Safety verification
   ├─ Style compliance
   └─ Integration testing

3. Feedback
   ├─ Suggestions for improvement
   ├─ Required changes
   └─ Approval or rejection

4. Merge
   ├─ Squash or merge commit
   ├─ Update CHANGELOG.md
   └─ Close related issues
```
